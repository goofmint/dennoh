# dennoh 要件定義 (Stage 1: Requirements)

> **対象バージョン**: v0.1（MVP）
> **作成日**: 2026-06-12
> **最終更新**: 2026-06-12（v0.1 MVP をメモ保存・FTS検索・CLI・stdio MCP に縮小、HTTP/REST/SSE を削除、ID はファイル名 UUID に統一）
> **ステータス**: Draft（ユーザーレビュー待ち）

---

## 1. 製品概要

### 1.1 プロダクト名
`dennoh`（電脳）

### 1.2 コンセプト
**「保存先は常に1つ、整理はAIが行う」** をコアコンセプトとする、ローカルファースト・AI前提の個人知識管理システム。

従来のノートアプリが「保存時に人間が整理する」のに対し、dennoh は「保存後にAIが整理する」。ユーザーはフォルダ分け、タグ設計、ノートブック管理を行わず、すべての情報を単一の保存先へ投入する。

v0.1 では UI を持たず、**MCP サーバー（stdio）** と **CLI** を提供する。ユーザーは Claude Desktop / Cursor / VS Code / Windsurf 等の MCP クライアントから dennoh を操作する。ランタイムは **Bun** で統一し、v0.2 で追加予定の GUI も Bun ベースの **Electrobun** で構築する方針とする。

### 1.3 ユーザーが行う行為
1. メモを書く（MCPクライアント経由で渡す）
2. 検索する（MCPクライアントまたは CLI 経由で問い合わせる）
3. URLを貼る（v0.x 以降）
4. AIへ質問する（v0.x 以降）
5. ファイルを登録する（v0.x 以降）

### 1.4 差別化ポイント
- 保存先は常に1つ（分類UIが存在しない、ではなく「分類のためのデータ構造をユーザーが設計しない」の意）
- フォルダ不要 / タグ設計不要
- URL貼り付けだけで知識化（v0.x）
- AIによる自動関連付け（v0.x）
- AIによるトピックページ生成（v0.x）
- AI回答が情報アクセスの主手段（v0.x）
- ローカルファースト
- MCP Server を標準搭載（v0.1 時点で本体そのもの）
- ユーザー自身のLLM/Embeddingを利用可能（BYOK、v0.x）
- 内部 Git によるバージョン履歴（dennoh が自動で commit）

### 1.5 想定ユーザー
- 情報を分類したくない知的生産者
- 大量のURL/メモを継続的に収集する人
- 既に MCP 対応 AI ツール（Claude Desktop / Cursor / VS Code / Windsurf 等）を日常的に利用している人
- 自分のデータを自分の端末に置きたい人（ローカルファースト志向）

## 2. スコープ

### 2.1 v0.1（本要件定義のコア）
- **MCP サーバー（stdio のみ）**（save / search / update / delete / status 等の標準ツール群）
- **CLI**（同等操作 + インデックス管理 + 設定操作）
- テキストメモの作成・編集・削除
- Markdown ファイルとしてユーザー選択フォルダへ永続化
- グローバル設定ファイル / CLI コマンドによる設定（保存フォルダ、言語など）
- 全文検索インデックス（FTS）の自動更新
- 内部 Git 管理による履歴保持と巻き戻し

**想定クライアント**: Claude Desktop / Cursor / VS Code / Windsurf 等の MCP クライアント。UI（GUI）は v0.1 では提供しない。

### 2.2 v0.x ロードマップ（コンセプトに記載・本要件範囲外）
- **Electrobun ベースの GUI**（v0.2 想定、Bun ランタイム + システム WebView 構成。本要件の末尾に参考仕様を記載）
- URL の取り込みと要約
- 自然言語による AI 検索・回答（出典引用付き）
- 全文検索 + ベクトル検索のハイブリッド検索
- 原文 + 英訳の両方を索引化する検索拡張
- YouTube URL の字幕取得・要約
- Markdown チェックボックスからのタスク抽出と完了管理
- エンティティ抽出（Person / Company / Product / Topic 等）
- エンティティ間リレーションのAI自動生成
- AI生成トピックページ / エンティティページ（動的生成）
- ブラウザ拡張による Web ページ取り込み
- RSS、Gmail、Google Calendar、GitHub、Slack、Notion、X 連携

### 2.3 明示的に範囲外
- モバイルアプリ（iOS/Android）
- マルチユーザー / 共有 / コラボレーション機能
- クラウド同期サービスを dennoh 側で運用すること（ユーザーのiCloud/Dropbox等を使う前提）
- E2E 暗号化された独自ストレージ
- **HTTP API / REST API / SSE**（v0.1 では提供しない）
- **MCP サーバーの HTTP トランスポート / 外部公開 / リモートアクセス**（v0.1 では提供しない。stdio に限定）

## 3. 機能要件（v0.1）

### F-1: メモ管理
- **F-1.1** ユーザー（または MCP クライアント）は新規メモを作成できる。メモは Markdown 形式のテキストとして渡される。
- **F-1.2** メモを更新できる（ID 指定で本文置換）。
- **F-1.3** メモを削除できる（ID 指定）。
- **F-1.4** メモは1ノート1ファイルとして Markdown ファイルに保存される。
- **F-1.5** 各メモは YAML frontmatter で作成日時・更新日時・ソース種別・プロジェクト・タグ等を保持する。ID は frontmatter に持たず、ファイル名の UUID を唯一の ID とする。
- **F-1.6** メモ本文中の `#プロジェクト名` は「プロジェクト」、`@タグ名` は「タグ」として **正規表現で機械的に抽出** し、検索・関連付けに利用する。AI 抽出ではなく決定論的な処理とする。
  - 例: 本文に `これは #denno プロジェクトの調査 @ai-coding` と書かれていれば、プロジェクト=`denno`、タグ=`ai-coding` として抽出
  - 抽出は保存時に同期実行し、frontmatter の `projects` / `tags` にミラーする
  - 正規表現パターン（暫定）: `#([\p{L}\p{N}_-]+)` / `@([\p{L}\p{N}_-]+)`（Unicode文字・数字・`_`・`-` を許容、確定は Stage 2）
  - URL の `#fragment`、Markdown 見出し `# H1`、メールアドレスの `@` を誤抽出しない判定ルールを実装する
- **F-1.7** 保存（=ファイル書き込み + インデクシング対象化）は MCP ツール `save_memory` を明示的に呼び出した時点で行う。自動保存・仮保存・編集中バッファのようなUI由来の概念は v0.1 では持たない。
- **F-1.8** frontmatter は dennoh が管理するメタデータ領域とし、ユーザーが編集する対象とはしない。外部エディタで本文を編集した場合、dennoh は再インデックス時に frontmatter を再生成・更新できる。

### F-2: 全文検索
- **F-2.1** 純粋な検索（AI 回答を伴わない取得）は MCP ツール `search_memory` / CLI `dennoh search` で提供する。
- **F-2.2** 検索は SQLite FTS を用いた全文検索とする。ベクトル検索・AI回答・英訳インデックスは v0.1 では実装しない。
- **F-2.3** 検索結果にはノート ID（ファイル名 UUID）、ファイルパス、タイトル、本文スニペット、作成日時、更新日時、source、projects、tags を含める。
- **F-2.4** 検索フィルタとして `#プロジェクト` / `@タグ`、日付範囲、source（note）を利用できる。
- **F-2.5** `list_recent` / `dennoh recent` により最近のノート一覧を取得できる。

### F-3: 永続化（Markdown ファイル）
- **F-3.1** ユーザーは保存先フォルダを設定で指定する（初回起動時は CLI セットアップウィザード `dennoh init` で必須入力）。
- **F-3.2** ノートはユーザー選択フォルダ以下の **日付ベースのサブディレクトリ**（例: `YYYY/MM/DD/`）に `<UUID>.md` として配置される。
- **F-3.3** ファイル書き込みはアトミック（一時ファイルへ書き込み → rename）。
- **F-3.4** ノートのメタデータは YAML frontmatter として埋め込む（createdAt, updatedAt, source, title, projects, tags 等）。`id` は frontmatter に含めない。
- **F-3.5** ユーザーはアプリ外でファイルを直接編集してもよい（外部編集を許容、変更検知して再インデックス）。
- **F-3.6** ID は UUID（v7 想定、時間ソート可能）。
- **F-3.7** ID はファイル名 `<UUID>.md` から決定する。frontmatter とファイル名の ID 不整合という状態は作らない。
- **F-3.8** 保存フォルダ内の dennoh 管理データは `<保存フォルダ>/.dennoh/` 配下に配置する（SQLite DB、キュー状態、ログ等）。

### F-4: 同期対応 / バージョン管理
- **F-4.1** iCloud Drive / Dropbox / OneDrive 等のクラウド同期フォルダを保存先に指定できる。
- **F-4.2** dennoh は保存フォルダを監視し、外部からの変更を検知してインデックスを差分更新する。
- **F-4.3** 同期ツールが生成する競合ファイル（例: `*.conflict.md`、`* (Atsushi's conflicted copy).md`）を検知してログ / `status` に記録する。競合の自動マージは v0.1 では行わない。
- **F-4.4** 隠しファイル（`.DS_Store`、`.git`、`.obsidian` 等）はインデックス対象外とする。
- **F-4.5** dennoh は保存フォルダを **自動的に Git 管理する**。保存フォルダが既存 Git リポジトリの場合はそのリポジトリを利用し、Git リポジトリでない場合は保存フォルダ直下に `.git` を作成する。
  - 保存ごとに自動 commit（コミットメッセージは `add <id>` / `update <id>` / `delete <id>` 形式）
  - ユーザーは Git の存在を意識する必要はない
  - 過去バージョンへの巻き戻しを CLI `dennoh history <id>` / `dennoh restore <id> <commit>` で提供
- **F-4.6** クラウド同期フォルダを保存先にした場合、`.git` ディレクトリは同期から除外することをセットアップ時に推奨表示する（`.dropboxignore` / `.nosync` 等の自動生成は v0.1 では行わない、案内のみ）。

### F-5: 設定
- **F-5.1** 設定可能項目:
  - 保存フォルダパス
  - 言語（`ja` / `en`）
- **F-5.2** macOS ではグローバル設定ファイルを `~/Library/Application Support/dennoh/config.json` に配置する。v0.1 ではこのファイルに保存フォルダパス等の非機密設定のみを保存する。
- **F-5.3** 保存フォルダ内の派生データ・状態データは `<保存フォルダ>/.dennoh/` 配下に保存する。
- **F-5.4** 設定操作は (a) 設定ファイル直接編集、(b) CLI コマンド `dennoh config get/set/list` の両方をサポートする。

### F-6: インデックス
- **F-6.1** dennoh 起動時に保存フォルダをスキャンし、ファイル変更を検知してインデックスを差分更新する。
- **F-6.2** ノート追加・更新時、原文本文とメタデータを SQLite FTS インデックスへ登録する。
- **F-6.3** インデックスは派生データであり、削除されても保存フォルダ（Markdown 群）から再構築可能。
- **F-6.4** CLI `dennoh reindex` でインデックスを手動再構築できる。
- **F-6.5** インデクシング中も MCP/CLI リクエストはブロックされない。

### F-7: MCP サーバー
- **F-7.1** dennoh は MCP サーバーとして起動でき、Claude Desktop / Cursor / VS Code / Windsurf 等の MCP クライアントから接続できる。
- **F-7.2** トランスポートは **stdio のみ**。Streamable HTTP、SSE、REST API は v0.1 では提供しない。
- **F-7.3** `dennoh serve` は MCP stdio サーバーとして起動する。stdout は MCP JSON-RPC 専用とし、ログ・診断情報は stderr に出力する。
- **F-7.4** v0.1 で提供する MCP ツール（最小セット）:
  - `save_memory(content, source?)` → ノート保存
  - `update_memory(id, content)` → 既存ノート更新
  - `delete_memory(id)` → ノート削除
  - `search_memory(query, filters?, limit?)` → 検索結果一覧（AI回答なし）
  - `list_recent(limit?)` → 最近のノート一覧
  - `get_note(id)` → ノート取得
  - `status()` → インデックス状態・キュー状態の取得
- **F-7.5** ツールのスキーマは MCP 標準仕様に準拠する。

### F-8: CLI
- **F-8.1** CLI コマンド名は `dennoh`。
- **F-8.2** 最小コマンド:
  - `dennoh init` → 初回セットアップ（保存フォルダ選択、設定生成、git 初期化）
  - `dennoh serve` → MCP stdio サーバーを起動
  - `dennoh add "<text>"` → メモを保存（stdin パイプ対応）
  - `dennoh update <id> "<text>"` → メモを更新（stdin パイプ対応）
  - `dennoh delete <id>` → メモを削除
  - `dennoh search "<query>"` → 検索
  - `dennoh get <id>` → ノート取得
  - `dennoh recent` → 最近のノート一覧
  - `dennoh status` → インデックス状態・キュー状態の確認
  - `dennoh reindex` → インデックス再構築
  - `dennoh history <id>` / `dennoh restore <id> <commit>` → バージョン履歴 / 巻き戻し
  - `dennoh config get/set/list` → 設定操作

## 4. 非機能要件

### 4.1 ローカルファースト
- ネットワーク接続が無くても、メモ保存・原文の全文検索は動作する。
- v0.1 の中核機能はローカルファイルと SQLite FTS のみで完結する。

### 4.2 セキュリティ / プライバシ
- v0.1 では API キーを扱わない。
- Markdown ファイル本体は平文保存（ユーザーがフォルダ自体を管理する前提）。
- MCP は stdio のみで提供し、HTTP エンドポイントは持たない。

### 4.3 パフォーマンス目安
- 1万ノート規模で:
  - 全文検索: < 200ms
- MCP サーバー起動: コールドスタート < 3秒（既存インデックスあり）

### 4.4 移植性
- 検索エンジンは将来のモバイル（iOS / Android）移植を**前提**とする。
- 具体的には SQLite FTS5 を優先候補とし、ネイティブバイナリ依存を最小化する。
- v0.x で Electrobun ベースの GUI を追加する際、v0.1 の core モジュールを再利用する設計とする。

### 4.5 信頼性
- ファイル書き込みは原則アトミック。
- 内部 Git により全変更がコミットされるため、任意時点へ巻き戻し可能。
- インデックス破損時は自動再構築で復旧（Markdown ファイル群を一次データとする原則）。

### 4.6 対応プラットフォーム
- v0.1 の優先対応は macOS（Apple Silicon / Intel）とする。
- Windows 10/11（x64）および Linux（x64）は v0.x で段階的に対応する。

### 4.7 国際化
- v0.1 から国際化を前提とする。
- 対象: CLI メッセージ、エラーメッセージ、MCP ツールの description。
- 既定言語: 日本語。最低限 `ja` / `en` を提供。
- 言語選択は `dennoh config set lang <ja|en>` または環境変数 `DENNOH_LANG`。

## 5. データモデル（要件レベル）

### 5.1 ストレージ階層
- **一次（Source of Truth）**: Markdown ファイル（ユーザー選択フォルダ）+ 内部 Git リポジトリ
- **二次（派生）**: `<保存フォルダ>/.dennoh/` 配下の SQLite データベース（全文検索インデックス、メタデータキャッシュ）
- **二次が消えても、一次から再構築可能** であることを設計原則とする。

### 5.2 Markdown ファイル形式
```markdown
---
createdAt: 2026-06-12T10:30:00+09:00
updatedAt: 2026-06-12T10:35:00+09:00
source: note
title: （任意、未指定なら本文先頭から自動）
projects: [denno, blog]                 # 本文中の #xxx から正規表現で抽出
tags: [ai-coding, mcp]                  # 本文中の @xxx から正規表現で抽出
---

本文（Markdown、`#プロジェクト` `@タグ` を含む）
```

### 5.3 ID とファイル名規則
- ID: ファイル名の UUID v7（時間ソート可能）
- ファイル名: `<UUID>.md`
- 配置: `<保存フォルダ>/<YYYY>/<MM>/<DD>/<UUID>.md`
- frontmatter には `id` を含めない。

### 5.4 v0.x で拡張する概念（v0.1 では実装しない）
- Entity（Person / Company / Product / Topic / Event）
- Task（チェックボックスから抽出）
- Relation（エンティティ間の関連）
- これらも最終的には Markdown ファイルまたは frontmatter で表現する方針とし、SQLite はあくまでインデックス層であり続ける。

## 6. クライアント体験要件（v0.1: MCP / CLI）

### 6.1 MCP クライアント体験
- Claude Desktop の設定で `dennoh serve` を stdio MCP サーバーとして登録するだけで利用可能。
- ユーザーは Claude Desktop 等から自然言語で「これを保存して」「#denno のメモを探して」と話すだけで dennoh が呼ばれる。
- LLM がツール呼び出しの判断を行うため、ユーザー側に明示的なツール選択 UI は不要。

### 6.2 CLI 体験
- 一般的な Unix CLI 慣習に従う（`--help`、`--json` 出力、終了コード）。
- パイプ可能（例: `cat memo.md | dennoh add`）。

### 6.3 状態表示
- バックグラウンド処理の進捗（インデクシング中など）は CLI の進捗表示または `dennoh status` で確認可能。
- `dennoh status` でキュー残数・現在処理中ジョブを確認可能。

## 7. 受け入れ基準（v0.1）

以下が満たされた時点で v0.1 リリース可能とする。

1. **MCP 経由のメモ保存**: Claude Desktop から `save_memory` を呼び出すと、保存フォルダに Markdown ファイルが生成され、再起動後も復元できる。
2. **CLI 経由のメモ保存**: `dennoh add` で Markdown メモを保存でき、stdin パイプにも対応する。
3. **FTS検索**: `search_memory` / `dennoh search` が SQLite FTS により保存済みメモを検索できる。
4. **#/@ 認識**: 本文に `#denno @mcp` と書かれたノートを保存後、`search_memory(filters: { project: "denno" })` で取得できる。
5. **ID規則**: ノート ID はファイル名 UUID と一致し、frontmatter に `id` は含まれない。
6. **外部編集反映**: 保存フォルダ内のファイルを他のエディタで編集すると、dennoh が再インデックスし反映する。
7. **同期共存**: iCloud Drive 配下に保存フォルダを設定しても、メモ作成・閲覧・検索が正常に動作する（`.git` の同期除外を案内する）。
8. **Git 履歴**: ノートを複数回更新したあと、`dennoh history <id>` で履歴を確認し、`dennoh restore <id> <commit>` で過去バージョンに戻せる。
9. **インデックス再構築**: `dennoh reindex` でインデックスを削除→再構築でき、全データが復元される。
10. **CLI 動作**: `dennoh init / serve / add / update / delete / search / get / recent / status / config / reindex / history / restore` が動作する。
11. **macOS動作**: macOS でビルド・起動・基本機能の動作確認が完了している。

## 8. 配布形態

### 8.1 ランタイム方針（v0.1 / v0.2 一貫）
- **Bun を公式ランタイムとする**。Node.js は明示的にサポートしない（互換動作する場合があっても要件外）。
- 理由: v0.2 で採用予定の Electrobun が Bun ランタイム前提であり、v0.1 から Bun に揃えることで v0.2 への移行コストを最小化する。
- Bun の対応バージョンは Stage 2 で確定（執筆時点での最新安定版を基準）。

### 8.2 v0.1 配布
- **Bun スタンドアロン実行ファイル**（`bun build --compile` で生成）として macOS 用バイナリを優先配布
- 補助的に **npm パッケージ**（`bunx dennoh`）も提供
- 配布物に Bun ランタイムを含む（ユーザー環境への依存なし）

### 8.3 v0.2 配布（参考）
- **Electrobun ベースの GUI** を追加し、`.app` / `.exe` / `.AppImage` 形式で配布
- Bun ランタイム + システム WebView（WKWebView / WebView2 / WebKitGTK）+ Zig ネイティブバインディングで構成
- バンドル内に dennoh の core / MCP サーバー / CLI を同梱（v0.1 のコードベースをそのまま利用）
- Electron 比 5〜6倍小さい配布物サイズが期待できる（Electrobun 公称値）
- ただし Electrobun は 2024年末リリースの若いプロジェクトであり、本番採用前に Stage 2 で安定性・3OS 対応状況・エコシステム成熟度を検証する

## 10. 用語

| 用語 | 定義 |
| --- | --- |
| ノート | 1つの Markdown ファイル単位の情報。v0.1 ではメモを扱い、v0.x 以降で URL 取り込み等も同じノートとして扱う。 |
| 保存フォルダ | ユーザーが選択したノート保存先のディレクトリ。dennoh が `.git` を作成し自動 Git 管理する。 |
| エンティティ | v0.x で導入する概念。Person/Company/Product/Topic 等の抽象概念単位。 |
| プロジェクト | 本文中の `#xxx` で表現される、複数ノートを束ねる軸（UIではなく本文記法）。 |
| タグ | 本文中の `@xxx` で表現される、ノートを跨ぐ横断的なラベル（UIではなく本文記法）。 |
| インデックス | SQLite 上に構築される派生データ。v0.1 では FTS とメタデータキャッシュを扱う。 |
| BYOK | Bring Your Own Key。v0.x でユーザーが自身のLLM/Embeddingプロバイダの APIキーを持ち込む方式。 |
| MCP | Model Context Protocol。dennoh が v0.1 から Server として標準提供。 |

## 11. v0.x クライアント UI 参考仕様（参考: 本要件の対象外）

> 以下は **v0.2 以降の Electrobun ベース GUI を想定した参考情報** であり、v0.1 要件には含まれない。core / MCP サーバー / CLI の設計時に「将来こうした UI を被せる」前提として参考にする。

### 11.1 主画面レイアウト案
- メモエリア（Markdown ビジュアルエディタ、画像ドロップ対応）
- コマンドパレット（Cmd+P 起動）
  - 既存ファイル検索
  - LLMチャット
- 保存ボタンは持たず、MCP の `save_memory` 相当を内部で呼び出す（コマンドや明示アクション経由）

### 11.2 入力体験案
- URL 貼り付けで自動的に取り込みフロー起動
- 通常テキストとURLの判別は入力内容の自動解析

### 11.3 検索体験案
- コマンドパレットを「質問入力欄」として利用
- パレット下に検索結果ファイル一覧、選択で編集画面へ
- Enter（return）で LLM 回答表示
- 引用元ノートは回答中にインライン表示

### 11.4 設定画面案
- 別ウィンドウまたはモーダル
- LLM/Embedding 設定、保存フォルダ変更、インデックス再構築

### 11.5 状態表示案
- バックグラウンド処理の進捗（インデクシング中、URL取り込み中、英訳生成中など。URL/英訳は v0.x）をステータスバーに表示
